## Original Intent

Extend the `/backlog` system so that issues receive intelligent enrichment scaled to their complexity. Simple issues get a clean title. Medium issues get auto-generated acceptance criteria and affected-file identification. Complex issues get deep-research findings and PRD skeletons attached to the issue body.

## Problem Statement

Issues created via `/backlog` receive a minimal template (title + "TBD" acceptance criteria). The brief interview post-creation catches some detail, but medium and complex issues need more: affected files, concrete acceptance criteria, research findings, PRD skeletons. Today this enrichment is entirely manual -- the user must invoke `/deep-research` and `/prd` separately, then manually paste findings back into the issue. This friction means most issues stay sparse, and implementer agents lack context when picking up work.

## Goals & Non-Goals

### Goals
- REQ-GOAL-001: Backlog issues get actionable enrichment (acceptance criteria, affected files, research) without manual effort
- REQ-GOAL-002: Enrichment depth scales with issue complexity automatically via classification
- REQ-GOAL-003: The pipeline is cost-safe -- no runaway API calls; complex enrichment requires confirmation

### Non-Goals
- REQ-NOGO-001: No ML-based classification -- keyword heuristics are sufficient for v1
- REQ-NOGO-002: No automatic deep-research on issue creation -- too slow (2-10 min) and expensive for fire-and-forget
- REQ-NOGO-003: No enrichment of issues outside the `/backlog` system (manually created GitHub issues)

## Requirements

### Must-Have (P0)
- REQ-P0-001: `classify_complexity(title, body) -> simple|medium|complex` function in `scripts/enrich-classify.sh`
  Acceptance:
  - Given title "Fix typo in README", returns "simple"
  - Given title "Add rate limiting middleware", returns "medium"
  - Given title "Redesign auth architecture for multi-tenant", returns "complex"
  - Given body >500 chars with architecture keywords, returns "complex" regardless of title

- REQ-P0-002: `/backlog enrich #N` command that reads an issue, classifies it, runs the appropriate enrichment pipeline
  Acceptance:
  - Given issue #50, when `/backlog enrich 50` runs, then issue body is updated with enrichment appropriate to its tier
  - Given an already-enriched issue (has `enriched` label), then warns and skips unless `--force` is passed

- REQ-P0-003: Medium enrichment -- auto-generate acceptance criteria + identify affected codebase files
  Acceptance:
  - Issue body contains an "Affected Files" section with file paths found via Grep/Glob
  - Issue body contains acceptance criteria bullets derived from the issue description
  - Both sections are generated by Claude's intelligence in the slash command flow

- REQ-P0-004: Complex enrichment -- invoke `/deep-research` and generate PRD skeleton, attach to issue
  Acceptance:
  - Issue body contains a "Research Summary" section with key findings
  - Issue body contains a "PRD Skeleton" section with problem statement, goals, requirements outline
  - User is prompted for confirmation before invoking deep-research (cost guard)

- REQ-P0-005: Issue body updated in-place, preserving original content
  Acceptance:
  - Original issue body appears under `<details><summary>Original Issue</summary>` block
  - Enriched content is prepended above the original
  - No original text is lost

- REQ-P0-006: `enriched` label added on completion
  Acceptance: After enrichment, `gh issue view` shows the `enriched` label

- REQ-P0-007: Rate limiting -- max 3 complex enrichments per hour, confirmation prompt
  Acceptance:
  - Given 3 complex enrichments already ran this hour, when user runs enrich on a complex issue, warning shown and confirmation required
  - Log stored in `~/.config/cc-todos/enrich-log.tsv`

### Nice-to-Have (P1)
- REQ-P1-001: At issue creation time, classify and suggest `/backlog enrich #N` for medium+ issues
- REQ-P1-002: Batch enrichment -- `/backlog enrich --all` to enrich all un-enriched issues (with confirmation)

### Future Consideration (P2)
- REQ-P2-001: Auto-enrich via PostToolUse hook on issue creation (non-blocking, background)

## Definition of Done

All P0 requirements satisfied. `/backlog enrich #N` works for simple, medium, and complex issues. Rate limiting prevents runaway costs. Existing `/backlog` functionality is unbroken.

## Architectural Decisions

- DEC-ENRICH-001: Enrichment orchestration in `backlog.md` slash command, not a standalone bash script -- because generating acceptance criteria and PRD skeletons requires Claude's intelligence. Bash handles classification and rate-limit checking only. Addresses: REQ-P0-002, REQ-P0-003, REQ-P0-004
- DEC-ENRICH-002: Keyword-based complexity classification in bash. "Simple" = fix/typo/config/bump/rename/update keywords + short body (<100 chars beyond template). "Complex" = architecture/redesign/multi-/migrate/security/compliance keywords OR body >500 chars with multiple concerns. Everything else = "medium". Addresses: REQ-P0-001, REQ-NOGO-001
- DEC-ENRICH-003: Rate limiting via timestamp file `~/.config/cc-todos/enrich-log.tsv` (timestamp, issue#, tier, duration). Checked before complex enrichment. Addresses: REQ-P0-007
- DEC-ENRICH-004: In-place body update wraps original in `<details>` block, prepends enriched content. Uses `gh issue edit --body`. Addresses: REQ-P0-005

**Research gate:** Skipped. All decisions use well-understood tools (gh CLI, bash heuristics, existing skills). No unfamiliar domains.
